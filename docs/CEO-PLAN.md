## 핵심 한 줄
"지라 같은 내 전용 칸반"이 아니라 **자율주행 프로젝트 매니저 + 인간 에스컬레이션 밸브**.
태스크 추적의 단일 진실 원천(GitHub 이슈 폐기, 이중화 無). LLM은 100% 로컬, 서버는 LLM 없이 상태·UI·알림만.

## Vision

### 10x Check
"더 좋은 칸반"이 아니라 **에이전트 조직도**. 당신 = CEO, 에이전트 = IC, 보드 = 스탠드업,
텔레그램 = 받은편지함, QA/CI = 품질 게이트. 태스크 관리가 아니라 **결정 관리**를 하게 됨.

### Platonic Ideal
밤새 로컬 에이전트가 티켓을 처리 → 아침 텔레그램 다이제스트 "12개 끝, 3개 결정필요" →
사이트 결정 큐에서 탭으로 승인 → 에이전트 재개. 카드 클릭 시 "왜 그렇게 했는지" 풀 타임라인.
배포만 무조건 허락. 스킬(jissue/jpr/scoophub-ship)과 CI는 나중에 이 시스템의 '액션'으로 흡수.

## Section 1: Architecture

### 시스템 아키텍처

```
  LOCAL (Mac)                              con-jjong (Oracle) · docker-compose
  ──────────                               ─────────────────────────────────────
  launchd/cron                             ┌─────────────────────────────┐
  (POLL_INTERVAL_SEC, 0=off) 구동           │ caddy (TLS 역프록시)         │
       ▼  headless                         │  ├─ web UI (nginx static)    │
  Claude Code agent ◀── 코드·PR             │  └─ Go API (token auth) ◀── 유일한 진입점
  (LLM 100% 여기)   │                       │       │   (DB 직접 접근 無)   │
       │            │ HTTPS + API token      │       ▼                      │
       │            │ pull(폴링)/push(결과·게이트결정요청)                    │
       │            └───────────────────────▶│  PostgreSQL (별도 DB: reins)  │
       │                                     │   · projects/tickets          │
       ▼                                     │   · ticket_events(append,커서) │
  GitHub repos ◀ 코드·PR                     │   · decisions                 │
  (scoophub 등)                              │   · agent_runs(heartbeat_at,  │
                                             │       lifecycle)             │
                                             │   · workflow_actions         │
                                             │   Telegram bot (outbound-only)│
                                             │   Watchdog 고루틴 + 다이제스트│
                                             └─────────────────────────────┘
                                                      │ 텔레그램 핑 (outbound만)
                                                      ▼ (결정/정보/아침다이제스트)
                                                당신 폰 → 웹 결정 큐 → 승인/거부
```
> D12-A 수정: 에이전트↔서버는 **전부 Go API 경유**(DB 직접 접근 無, 보안), postgres는 **별도 `reins` DB**(D3), Telegram은 **outbound-only**(D5, 공용 웹훅 無).

### 상태 기계 (tickets.status)
```
                 start                ready_for_qa            approve
   todo ──────────────▶ progressing ──────────────▶ qa ──────────────▶ done (terminal)
                            │                          │
                            │  (게이트: PR/머지/배포/    │  reject / changes_requested
                            │   스펙애매 → decision)     ▼
                            │                       progressing
                            ▼
                       [대기/재개]                any ──cancel──▶ cancel (terminal)
                  (decision pending)            done/cancel = terminal, reopen은 명시
```
- **불가 전이 방지**: API 레이어에서 화이트리스트 검증. done→todo 등은 `reopen` 엔드포인트로만.
- **레이스 방지**: 에이전트 티켓 클레임 = `UPDATE tickets SET status='progressing', agent_run_id=X WHERE id=:id AND status='todo' RETURNING` (원자적). 두 에이전트 동시 → 1명만 성공.

### 워크플로우/액션 엔진 (핵심 추상화)
```
  trigger(전이/이벤트) ──▶ ActionRegistry.lookup ──▶ [agent_run|telegram|decision|deploy|ci]
                                                       │
                                          v1 기본: todo→prog = agent_run
                                                  any 게이트 = decision (+telegram)
                                                  any 전이 = telegram(정보성)
                                          Phase2: prog→qa = ci, done = deploy(게이트)
```
- v1은 하드코딩 기본 규칙 + `workflow_actions` 테이블로 확장점만 열어둠. **과잉 추상화 금지**(ponytail): 인터페이스·팩토리 없이, 타입별 switch로 시작.

### 커플링·SPOF
- 서버 단일 인스턴스 = SPOF이지만 개인 도구 허용 범위. postgres 백업으로 복구.
- 로컬 에이전트는 휘발성 — 상태는 전부 서버 DB에 있으므로 로컬 크래시 = 재시작 시 큐에서 재개.

## Section 2: Error & Rescue Map

| 코드패스 | 실패 | 예외/상태 | 구조? | 액션 | 사용자 노출 |
|---|---|---|---|---|---|
| 에이전트 티켓 클레임 | 동시 2개 | 경쟁(1명만 RETURNING) | Y | 패자는 다음 티켓 | 없음 |
| 로컬 에이전트 크래시 | 런 중단 | `agent_run` stuck | **heartbeat watchdog** | 30분 무업데이트 → failed + 텔레그램 | 결정핑 "티켓X 정체" |
| GitHub PR 오픈 | timeout/429/auth | API 에러 | Y | backoff 재시도 → 실패 시 agent_run failed | 텔레그램 |
| 결정 이중 처리 | 중복 승인 | idempotent 위반 | Y | `UNIQUE` + `status='pending'` 재확인 | 없음 |
| Telegram 전송 실패 | 봇 다운/429 | Send 에러 | Y | 재시도 → event log 백업 | 웹 타임라인에 노출 |
| 에이전트 결과 파싱 | malformed/empty | 스키마 위반 | Y | 검증 실패 → agent_run failed, 카드 이동 無 | 텔레그램 |
| DB 마이그레이션 | 잠금/실패 | tx rollback | Y | 롤백, 구버전 유지 | 배포 중단 |
| 결정 만료 방치 | pending 방치 | 노화 | Y | 노화 알림(1h/24h) | 텔레그램 리마인드 |

**CRITICAL GAP 방지**:
1. 에이전트 "running" 영구 정체 → **heartbeat watchdog (P1, 필수)**. 없으면 카드가 보이지 않게 멈춤.
2. 결정 중복 해결 → **idempotent resolve + UNIQUE 제약 (P1)**.
3. Telegram 다운 → **event log 폴백 (P1)**. 알림이 안 가면 웹이라도 보여야.
4. 카드 상태 레이스 → **원자적 클레임 (P1)**.

> LLM 결과는 로컬에서 생성되지만, **서버가 결과 스키마를 검증**해야 함 — malformed면 카드를 조용히 옮기지 않음(신뢰).

## Section 3: Security & Threat Model

자산: 개인 태스크 데이터, GitHub 연동 토큰(로컬 only), Telegram 봇 토큰(서버).

| 위협 | 가능성 | 영향 | 완화(v1) |
|---|---|---|---|
| 서버 침해 → 데이터 노출 | Med | Med | **LLM/코드 실행 無** → 파급 최소화. prod secret 서버 無 |
| API 토큰 탈취 | Med | High | 강한 랜덤 토큰, env 보관, 회전 가능, **Tailscale로 관리포트 격리** |
| 웹 UI 무단 접속 | Med | Med | 단순 인증(비밀번호/토큰 쿠키) + Tailscale 권장 |
| SQL 인젝션 | Low | High | Go `database/sql` 파라미터화 쿼리. 문자열 결합 금지 |
| 입력 검증 부족 | Med | Med | 모든 API 입력 검증(제목 길이, status enum, id)·loud reject |
| Telegram 웹훅 위장 | Low | Low | `secret_token` 검증 |
| 무제한 요청(DoS) | Low | Low | 공개 엔드포인트 rate limit |

**권장 아키텍처 (Tailscale)**:
- 관리/에이전트/웹 API = Tailscale 망 안에서만. 공용 노출 최소화.
- 공용 = Telegram 웹훅(또는 long-polling으로 웹훅 자체 제거) + 읽기 전용 보드(선택).
- HTTPS 필수(Caddy 자동 TLS).

> 서버가 LLM/코드를 안 돌리는 결정(D5)이 보안 모델의 가장 큰 자산. 이 원칙을 지키세요.

1. **Board** — **단일 프로젝트 스코프**(사이드바에서 프로젝트 전환, 헤더엔 현재 라벨만; 한 칸반=한 프로젝트, 한 프로젝트 내 다중 에픽). 컬럼 todo/**progressing**/qa/done/cancel(풀 단어, 약어 금지). **그룹 드롭다운**: 없음(flat)/에픽별(동일 칸반 디자인+에픽 헤더 분할, "에픽 없음" 섹션). **정렬 드롭다운**: 등록순(기본)/최근변경순/우선순위. **에픽 카드**(indigo 보더+진행도바, 자식 done/total). 카드 필드: ID·agent 상태 배지(running/idle/failed)·결정필요 인디케이터·코멘트 수(단일 프로젝트 보드라 카드에 프로젝트 표시 생략).
2. **DecisionQueue** — 4게이트(PR오픈·머지·배포·스펙애매) 결정 카드. Approve(slate-900)/Reject/Changes, 더블클릭 idempotent(fade-out+토스트). **노화 강조(D-DR5)**: elapsed 표시, 1h+ amber, 24h+ red.
3. **TicketForm** — type(Task/Epic)·프로젝트·**에픽 셀렉트**(parent_id 지정)·우선순위·설명. 필드별 인라인 에러 + `aria-live`.
4. **Timeline** — 카드별 이벤트 로그(append, mono) + **Comments 탭**(수동 코멘트 리스트+입력, **agent-read ✓ 인디케이터** D-DR6). 커서 페이지네이션.

**상태 매핑**: 5컴포넌트(Board/DecisionQueue/TicketForm/Timeline/Comments) × loading/empty/error/success/partial — [`DESIGN-SYSTEM.md`](designs/DESIGN-SYSTEM.md) 표. 빈 상태 = 따뜻한 메시지 + 1차 액션(콜드스타트 F3: "첫 프로젝트 만들기").
